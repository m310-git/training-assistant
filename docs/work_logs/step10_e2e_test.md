
# Step 10: E2Eテスト

## 概要

全コンポーネントを結合した状態で、主要なユーザーフローを手動テストする。

## 前提条件

- [ ] Step 5〜9 全て完了
- [ ] 全コンポーネントがローカルで動作確認済み

---

## 手順

### 10-1. テストデータのクリーンアップ

テスト用データを削除して、クリーンな状態からテストする:

    bq query --use_legacy_sql=false \
        "UPDATE raw.training_log SET is_deleted = TRUE, updated_at = CURRENT_TIMESTAMP() WHERE log_id LIKE 'test-%'"

    # dbt run で mart を更新
    cd dbt
    dbt run

### 10-2. E2Eテストシナリオ

#### シナリオ1: 新規トレーニング記録の入力

    テスト手順:
    1. Streamlitにuser_001でログイン
    2. Input画面を開く
    3. 日付: 今日、部位: 胸、種目: ベンチプレス を選択
    4. セット1: 80kg × 5回 RPE 8.0 を入力
    5. セット2: 85kg × 3回 RPE 9.0 を入力
    6. セット3: 90kg × 1回 RPE 9.5 を入力

    期待結果:
    - [ ] 各セットが自動保存される（✅マーク表示）
    - [ ] 総負荷量がリアルタイムで計算される
    - [ ] BigQuery raw.training_log に3レコードが追加される

#### シナリオ2: 筋トレ開始通知

    テスト手順:
    1. シナリオ1の初回保存時に自動でトリガーされる

    期待結果:
    - [ ] 他の2ユーザーにLINE通知が届く（仮LINE IDの場合はログで確認）
    - [ ] notification_log に start タイプのレコードが記録される
    - [ ] 同日2回目の記録では通知が送信されない

#### シナリオ3: 当日復元と編集

    テスト手順:
    1. シナリオ1の後、同じ日付・種目を再度選択
    2. 既存のセットデータが復元されることを確認
    3. セット1の重量を82.5kgに変更

    期待結果:
    - [ ] 「⚠️ 本日のこの種目の記録があります」が表示される
    - [ ] 既存の3セットが復元される
    - [ ] 編集が保存される（updated_at が更新される）

#### シナリオ4: 編集制限（3時間超過）

    テスト手順:
    1. 3時間以上前に作成したレコードの編集を試みる
    （テスト用にcreated_atを手動で過去に設定してもよい）

    期待結果:
    - [ ] 入力フィールドがdisabled（グレーアウト）になる
    - [ ] 編集ボタンが非表示になる

#### シナリオ5: dbt日次パイプライン

    テスト手順:
    1. Cloud Functions の dbt-runner を手動実行

        gcloud functions call dbt-runner \
            --region=asia-northeast1 \
            --data='{}'

    期待結果:
    - [ ] dbt run が成功する
    - [ ] staging.stg_training_log にデータが反映される
    - [ ] mart層の全テーブルが更新される
    - [ ] dbt test が全てPASS

#### シナリオ6: ダッシュボード表示

    テスト手順:
    1. Dashboard画面を開く
    2. KPIカードを確認
    3. 進捗グラフを確認
    4. 種目別グラフを切り替え

    期待結果:
    - [ ] KPIカードに複合種目の週次変化率が表示される
    - [ ] 進捗グラフが描画される
    - [ ] 種目別グラフが切り替わる
    - [ ] 期間フィルタが動作する

#### シナリオ7: カレンダー表示と編集遷移

    テスト手順:
    1. Calendar画面を開く
    2. トレーニングした日をクリック
    3. 詳細を確認
    4. 「✏️ 編集する」ボタンをクリック

    期待結果:
    - [ ] トレーニング日に🟢マークが表示される
    - [ ] 日付クリックで詳細が表示される
    - [ ] 3時間以内の記録に編集ボタンが表示される
    - [ ] 編集ボタンでInput画面に遷移する

#### シナリオ8: ランキング表示

    テスト手順:
    1. Ranking画面を開く
    2. 週間/月間/全期間タブを切り替え
    3. 部位別ランキングを確認

    期待結果:
    - [ ] 各タブでランキングが表示される
    - [ ] 順位変動（↑↓→NEW）が表示される
    - [ ] 部位別ランキングが動作する

#### シナリオ9: ソーシャル機能

    テスト手順:
    1. Social画面を開く
    2. 記録更新フィードを確認
    3. 他ユーザーの記録を閲覧

    期待結果:
    - [ ] 記録更新フィードが表示される（is_new=TRUEのレコード）
    - [ ] 他ユーザーの記録が閲覧できる（編集不可）

#### シナリオ10: 種目追加リクエスト→承認フロー

    テスト手順:
    1. user_002でログイン
    2. ExerciseRequest画面で「ディップス」をリクエスト
    3. user_001（管理者）でログイン
    4. Admin画面で承認

    期待結果:
    - [ ] リクエストが送信される（status=pending）
    - [ ] 管理者画面に承認待ちとして表示される
    - [ ] 承認後、exercise_masterに追加される
    - [ ] 次回dbt run後にd_exerciseに反映される
    - [ ] Input画面で新種目が選択可能になる

#### シナリオ11: ML提案表示

    テスト手順:
    1. Input画面で過去データのある種目を選択
    2. 提案が表示されることを確認

    期待結果:
    - [ ] MLデータがある場合:「🤖 AIモデルによる提案」が表示される
    - [ ] MLデータがない場合:「📈 過去実績ベースの提案」が表示される
    - [ ] 提案通りの総負荷量が表示される

#### シナリオ12: LINE通知（日次・週次・月次）

    テスト手順:
    1. 各Cloud Functionsを手動実行

        gcloud functions call notifier-daily --region=asia-northeast1 --data='{}'
        gcloud functions call notifier-weekly-ranking --region=asia-northeast1 --data='{}'
        gcloud functions call notifier-monthly-ranking --region=asia-northeast1 --data='{}'

    期待結果:
    - [ ] 日次通知: 条件に該当するユーザーに通知が届く
    - [ ] 週間ランキング: 全ユーザーにランキング通知が届く
    - [ ] 月間ランキング: 全ユーザーにランキング通知が届く
    - [ ] notification_log にログが記録される

#### シナリオ13: 通知抑制テスト

    テスト手順:
    1. notification_log に大量のダミーデータを投入して月間カウントを180以上にする
    2. 日次通知を実行

    期待結果:
    - [ ] 180通以上: 7day通知が suppressed になる
    - [ ] 190通以上: 3day通知も suppressed になる
    - [ ] 200通以上: 全通知が suppressed になる
